# 项目文件规范

本规范适用于 `src/pocket_app` 及其直接相关文件，目标是让项目结构稳定、职责清晰，并且在功能继续增长时仍然容易维护。

## 核心目标

- 目录先表达职责，再表达技术实现。
- 新增代码优先放到现有模块的合理位置，不随意制造“杂物文件”。
- 一个文件只承担一类主要职责；当职责开始混杂时，及时拆分。
- 让后来维护的人仅凭目录名和文件名，就能大致判断代码应该去哪里找。

## 当前项目根结构

- `src/pocket_app/`：应用主包，业务代码统一放在这里。
- `skills/`：仓库内的技能文档，不放运行时代码。
- `pyproject.toml`：项目元数据与依赖声明。

除非有明确理由，不要把应用代码直接放到仓库根目录，也不要绕开 `src/pocket_app/` 新建另一套平行代码目录。

## `src/pocket_app` 下的目录职责

- `main.py`：程序入口，只负责应用启动、顶层对象装配和主窗口拉起。
- `views/`：页面、窗口和交互编排逻辑，负责“界面如何组织与响应”。
- `components/`：可复用 UI 组件，负责被多个页面或窗口复用的界面片段。
- `styles/`：QSS 和样式相关代码或资源引用入口，负责视觉样式组织。
- `api.py`：远程接口访问、抓取、异步请求入口；当接口逻辑增长时可拆成 `api/` 包。
- `utils.py`：通用工具函数；当工具函数明显分组后可拆成 `utils/` 包。
- `__init__.py`：包导出入口，只暴露值得对外稳定引用的对象，不堆放业务实现。

## 文件应该放在哪里

### 入口与装配

- 应用启动代码放在 `main.py`。
- 创建 `QApplication`、主窗口实例、启动事件循环这类逻辑留在 `main.py`。
- 不要把具体页面实现、网络请求细节或大段数据处理写进 `main.py`。

### 视图层

- 窗口类、页面类、交互流程放在 `views/`。
- `views/` 中的代码可以协调组件、调用 API、触发状态更新，但不要把底层通用逻辑直接内联进去。
- 一个页面文件通常对应一个主要页面或窗口；当一个文件里出现多个相互独立的大块界面时，应考虑拆分。

### 组件层

- 会在多个页面重复出现的控件组合放到 `components/`。
- 组件负责自己的局部 UI 构建和有限交互，不承担页面级流程控制。
- 如果某段控件代码只在单个页面里出现一次，优先留在对应 `views/` 文件中，不要过度抽象。

### 样式层

- QSS 文本、样式加载函数、样式常量放到 `styles/`。
- 样式目录只处理视觉表现，不放业务逻辑。
- 如果后续引入多个主题，可在 `styles/` 下按主题或用途拆分文件。

### 数据访问与工具

- 请求接口、解析远端响应、异步抓取相关代码放到 `api.py` 或 `api/` 包。
- 与 UI 无关、可复用的纯函数工具放到 `utils.py` 或 `utils/` 包。
- 不要把只服务于单个窗口的小型私有辅助函数提前抽到 `utils.py`；只有跨模块复用或职责明确时再抽离。

## 何时从单文件拆成包

当下面任一情况出现时，优先考虑从单文件拆成目录包：

- 文件已经同时承载多类职责，例如既有请求逻辑又有数据转换又有缓存处理。
- 文件长度明显增长，阅读时需要频繁跳转定位不同职责。
- 某一类能力已经形成稳定分组，例如多个 API 客户端、多个工具子模块、多个样式主题。
- 新增功能继续堆进当前文件会让导入关系或命名变得混乱。

建议拆分方向：

- `api.py` -> `api/`
- `utils.py` -> `utils/`
- `styles/` 下的单一大文件 -> 按主题、控件类型或页面拆分

拆包后保留 `__init__.py` 作为统一导出入口，避免外部调用方依赖过深的内部路径。

## 命名与文件粒度

- Python 文件名使用 `snake_case`。
- 与页面或窗口对应的文件名应体现具体对象，例如 `main_window.py`、`pokemon_detail_view.py`。
- 不使用含糊文件名，如 `common.py`、`temp.py`、`misc.py`，除非内容边界非常明确且团队已达成一致。
- 一个文件优先围绕一个主要概念组织，不把互不相干的类和函数硬塞到一起。

## `__init__.py` 约束

- `__init__.py` 负责包级导出和极少量必要元数据。
- 可在这里整理对外暴露的类或函数，方便外部以稳定路径导入。
- 不在 `__init__.py` 中写复杂初始化、副作用逻辑或大量实现代码。

例如，`views/__init__.py` 适合导出 `MainWindow`，但不适合直接写窗口实现。

## 导入关系约束

- `main.py` 可以导入 `views/`，也可以装配 `api`、`styles` 等顶层依赖。
- `views/` 可以依赖 `components/`、`styles/`、`api.py`、`utils.py`。
- `components/` 可以依赖 `styles/` 和少量通用工具，但应尽量避免反向依赖具体页面。
- `utils.py` 保持底层通用，不依赖页面和窗口。
- `api.py` 不依赖具体页面类，避免把 UI 逻辑反向耦合进数据访问层。

如果出现循环导入，优先检查是不是文件职责拆分不合理，而不是用临时局部导入掩盖结构问题。

## 新增文件前的决策顺序

新增代码时按这个顺序判断放置位置：

1. 这是启动装配逻辑吗？如果是，放 `main.py`。
2. 这是页面或窗口交互逻辑吗？如果是，放 `views/`。
3. 这是可复用控件吗？如果是，放 `components/`。
4. 这是样式或主题组织吗？如果是，放 `styles/`。
5. 这是远程请求或接口访问吗？如果是，放 `api.py` 或 `api/`。
6. 这是与 UI 无关的通用函数吗？如果是，放 `utils.py` 或 `utils/`。
7. 如果以上都不完全匹配，先检查是否应该扩展现有模块，而不是立刻新建顶层目录。

## 与现有 Python 规范的关系

- 本规范解决“代码应该放在哪里”。
- `skills/python风格规范/skill.md` 解决“代码在文件里应该怎么写”。
- 两者同时适用时，先按本规范确定目录与文件，再按 Python 风格规范实现代码。

## 提交前自检

- 新增文件是否放在了 `src/pocket_app/` 的合适职责目录下。
- 文件名是否准确表达内容，而不是临时命名。
- `main.py` 是否仍然保持精简，没有膨胀成业务杂糅入口。
- 页面逻辑、可复用组件、样式、接口访问、工具函数是否被混放。
- 是否出现了本应拆包却继续堆在单文件中的趋势。
- `__init__.py` 是否只做导出，没有藏入复杂实现。
