# Python 风格规范

本规范适用于 `src/pocket_app` 下的所有 Python 代码。

## 核心目标

- 优先可读性，避免为了“简洁”牺牲理解成本。
- 优先明确性，名称、类型和职责边界都要清楚。
- 优先小步扩展，新增逻辑先放到合适模块，不把无关职责堆进同一个文件。

## 基本要求

- 为公开函数、方法、类属性和返回值补充类型标注。
- 模块名、文件名、函数名使用 `snake_case`。
- 类名使用 `PascalCase`。
- 常量使用 `UPPER_SNAKE_CASE`。
- 导入顺序按标准库、第三方、本地模块分组。
- 删除未使用的导入、变量和空壳占位代码。
- 默认使用 ASCII；只有文件本身已经使用中文或确有必要时再引入非 ASCII 文本。

## 函数与类

- 单个函数只做一件事；当一个函数同时处理 UI、数据转换和网络请求时，应拆分。
- 函数名要体现动作和结果，例如 `load_user_profile`、`build_stat_cards`，避免 `handle_data` 这种宽泛命名。
- 类要围绕单一职责组织。窗口类负责界面组装与交互协调，不要把复杂业务逻辑直接塞进窗口类。
- 简短且明显的代码不必写注释；只有在约束、顺序依赖或框架行为不直观时才加简洁注释。

## 数据与控制流

- 优先早返回，减少多层嵌套。
- 对可空值、空列表、网络失败和资源缺失这类边界情况做显式处理。
- 不要吞掉异常；如果捕获异常，就补上明确的恢复逻辑、提示信息或重新抛出。
- 能写成纯函数的工具逻辑，放到独立函数中，避免耦合 UI 状态。

## 这个项目中的具体约束

- `main.py` 只保留应用启动和顶层装配逻辑。
- `views/` 下的类负责窗口、页面和交互编排。
- 可复用的界面片段放到 `components/`，不要在多个窗口里复制同一段控件构建代码。
- 通用工具函数放到 `utils.py` 或后续拆出的 `utils/` 包中。
- 与远程接口、抓取或异步请求相关的逻辑放到 `api.py` 或后续拆出的 API 模块中。

## 提交前自检

- 新增或修改的函数是否有清晰类型标注。
- 文件名、类名、函数名是否符合命名规范。
- 代码是否把 UI、业务和工具逻辑混在一起。
- 是否留下了未使用导入、调试输出或空实现。
- 阅读一遍代码时，是否能仅凭名称和结构快速理解作用。
